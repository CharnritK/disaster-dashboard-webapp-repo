# 00 — Master Codex Prompt

## Role

You are Codex acting as a senior TypeScript/React engineer and visualization-quality implementation agent.

## Objective

Implement visualization guardrails for `CharnritK/disaster-dashboard-webapp-repo`, branch `001-decision-context-data-quality`, so dashboard outputs generated by the copilot are controlled by deterministic standards for chart selection, color, layout, titles, accessibility, mobile behavior, quality caveats, and geospatial suitability.

## Critical behavior

Use sub-agents or equivalent isolated work passes. Each sub-agent must inspect only the files needed for its task and return a compact handoff before the coordinator merges changes. This is required to optimize context window and reduce accidental broad rewrites.

If native sub-agents are unavailable, emulate them using sequential passes:
1. discover;
2. schema/policy;
3. rendering;
4. tests;
5. docs/skill;
6. final QA.

## Ground rules

- Do not rewrite the application.
- Do not replace the current charting stack unless the repo already has an approved charting dependency.
- Prefer custom React + SVG/HTML improvements because that matches the current repo context.
- Do not remove existing data pipeline behavior.
- Do not claim tests passed unless you run them and include command output.
- Do not add heavy dependencies without a clear reason and explicit note.
- Do not handle secrets or credentials.
- Do not deploy.
- Do not modify production data.
- Stop before destructive changes.

## Required feature outcome

The dashboard recommendation pipeline must produce or reconcile chart specs that satisfy visualization standards before rendering.

The feature must add:

1. A deterministic visualization policy module.
2. Extended chart spec fields for units, subtitles, annotations, quality badges, sorting, mobile behavior, and source/caveat notes.
3. Shared chart frame component for consistent title, subtitle, quality badge, source/caveat, and accessibility hooks.
4. Color and status tokens that are consistent, accessible, and semantically scoped.
5. Validation rules that reject or downgrade misleading charts.
6. Tests for policy, schema, rendering contract, accessibility basics, mobile behavior, and simulated disaster data.
7. A documentation/skill package that teaches the copilot the standards while deferring hard rules to code.

## Deterministic controls

Implement these in code, not prompt-only:

- Comparison/ranking defaults to bar or dot-style chart.
- Trend over time defaults to line only when at least two valid time points exist.
- Pie/donut is allowed only for part-to-whole, positive values, and small category count.
- Pie/donut with too many slices must become bar.
- Bar chart value axis starts at zero.
- Line chart sort is time ascending.
- Long labels use horizontal layout or truncation with full tooltip/label access.
- Choropleth uses rate/ratio/density only, never raw counts.
- Raw geospatial counts fall back to ranked bar, proportional symbol, or table until a map component supports them.
- Color is never the only encoding of status.
- Text contrast target is at least 4.5:1.
- Meaningful graphical objects and focus indicators target at least 3:1.
- High-severity quality issues produce `qualityBadge: "block"` or a rendered warning/caveat.
- Missing geometry for map specs must produce a non-map fallback.
- Dense charts must reduce marks or provide table fallback.

## Non-deterministic controls

The LLM/copilot may generate only:

- insight-driven chart titles;
- concise subtitles;
- annotation labels;
- explanatory rationale;
- source/caveat wording.

The LLM/copilot must not choose final chart type, color semantics, blocking status, map normalization, sort order, or accessibility requirements after `enforceVizPolicy()` runs.

If adding component/a11y/E2E tooling requires dependencies, make that a separate phase unless dependencies already exist.

## Stop conditions

Stop and ask for human review if:

- target type definitions conflict with existing public API;
- changes would require replacing the charting stack;
- map implementation requires a new large library;
- accessibility test tooling requires dependency approval;
- the dashboard output schema is used by external clients;
- tests cannot be run;
- there is any request for credentials, tokens, deployment, or production data changes.
